How ClaudeOS prioritizes tasks

Priority isn't a flag you set. It's a score the engine recomputes every 5 seconds from live signals — so the queue re-sorts itself as your sessions change.

The one-paragraph version

Every tick, the engine looks at all your Claude sessions. First it filters by state — only sessions that are genuinely ready for you (waiting on input or done) are allowed through; anything still working stays hidden. Then it scores the survivors with a transparent points formula and sorts them high→low. The top one is the recommendation. You shape the ranking by nudging the inputs (focus, importance, snooze, up/down) — never a raw priority number.

1 The tick: what runs every 5 seconds

One pass of engine.tick() — fired on a timer (tick_interval_ms: 5000) and again whenever you act on a card.

STAGE 1 · DISCOVER
Find recent sessions
Scan ~/.claude/projects/ for sessions that have moved lately.
STAGE 2 · DETECT STATE
Is each session ready for me?
Tail-read the transcript → WORKING / WAITING_INPUT / DONE / UNKNOWN. This is the gate.
STAGE 3 · TRIAGE
What kind of attention does it need?
Label ready ones: simple question · review a diff · complex decision · FYI done. (Rules first, cheap LLM if unsure.)
STAGE 4 · RANK ◀ the prioritization
Score every ready item, sort high→low
Weighted signals + learned nudges + your overrides → one number per item.
STAGE 5 · ENRICH
Add a one-line summary + suggested answer
Background claude -p so the card reads well. Doesn't affect rank.

Also fired by: any card action (snooze / nudge / answer → fast re-score) and a nightly "dream" ~03:00 that does the learning.

2 The gate comes first: state before priority

A task can only be prioritized if it's first judged ready. This filter runs before any scoring, and it's deliberately strict.

WAITING_INPUT SURFACED

Turn ended cleanly and the text asks something — ?, "would you like", "confirm", "ready for review" — or a needs input: marker.

DONE SURFACED

A result:/failed: marker, or text like done / finished / merged / deployed.

WORKING HIDDEN

Process alive & wrote within 4s, or last turn is a tool_use / tool_result — still mid-task. Never shown.

UNKNOWN LOW

Ended cleanly but no question and no done-marker → idle/parked. Surfaced but sunk by idle_base −8.

The hard safety rule: false-surface rate = 0.

Interrupting you about a session that's actually still working is the worst failure mode. The eval (src/eval/run.ts) measures this directly and fails the build if it's ever above zero. So when state is ambiguous, the answer is "stay hidden."

3 The score: a transparent points total

Every ready item gets one number from scoreItem(). Nothing is a black box — each item keeps a breakdown of exactly which signals gave it how many points.

score = Σ ( weight × signal ) + Σ learned nudges + your overrides

4 How you change a task's priority

You never type a raw priority number. You push the inputs and the score re-cooks itself on the next (instant) re-rank.

set focusDeclare your focusA text focus boosts every matching task via focus_match + steers the importance LLM.
importance 0–100Manual importanceOverrides the LLM's judgment for that one item.
↑ / ↓ nudgeRank higher / lowerA persistent per-item delta. Adding a typed reason teaches the nightly learner ~15× harder.
snoozeSnooze (−40)Sinks it toward the bottom but keeps it visible. Stacks (cap 3×); cleared on complete.
pinPin to topAdds PIN_BASE = 100000 — forced above everything.
open terminalActive task floatsOpening a session lifts it to just above the rest (+ACTIVE_OVER), readable numbers — not a flat 100k.
The nightly "dream" learns your taste.

~03:00 it reads how you actually triaged (which cards you picked, snoozed, nudged, and why) and writes per-category nudges + evolves the plain-English rules in config/RANKING.md. It never edits weights.json. Teaching strength is graded: a silent pick ≈ 0.1×, a bare ↑/↓ ≈ 5×, and a ↑/↓ with a typed reason ≈ 15× — so explaining "I don't want this because…" is by far the strongest signal.

5 How the queue is finally ordered

The UI just reads the scored items out of the DB, sorted:

ORDER BY priority DESC, updated_at DESC

So: highest score first, ties broken by most recently changed. Then if you've opened a session, that active task floats to #1. The first item becomes next — the recommendation shown in the focused pane — and the rest are the queue list.

∴ Mental model

It's an assistant triaging your inbox every 5 seconds with two firm rules: (1) never bug you about something still in progress, and (2) of the things that genuinely need you, sort by a transparent points total — how important, does it block, is there a deadline, is it on your focus, how quick to clear, how stale — adjusted by what it's learned from your past choices and any overrides you've made.

config/weights.json — signal weights, thresholds, cadence, learning rates
config/RANKING.md — human-readable learned ranking rules (evolved nightly)
src/eval/run.ts — the false-surface-rate gate (must be 0)
